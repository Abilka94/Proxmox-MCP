# ADR-0006: Двухуровневая модель знаний (Infrastructure + Service)

**Статус:** accepted  
**Дата:** 2026-06-03  
**Зависит от:** ADR-0005

## Контекст

Агенту нужно рассуждать как сисадмин: от «сервис недоступен» до проверки гостя, ноды и кластера. PVE API не знает о пользовательских «сервисах» (стек приложений). Требуется слой без привязки к брендам продуктов.

## Решение

Ввести **два уровня знаний**:

### Уровень 1 — Infrastructure Layer (Proxmox-native, SoT = PVE API)

Cluster, Node, LXC, VM, Storage, Network, Task, Backup, Update.

### Уровень 2 — Service Layer (SoT = Operator Memory)

Универсальная сущность **Service** с полями:

- **Name** — человекочитаемое имя инстанса  
- **Type** — категория из контролируемого словаря (ADR-0010), не имя продукта  
- **RunsOn** — `EntityRef` на LXC или VM  
- **Dependencies** — список `EntityRef` (Service или Infrastructure)  
- **HealthStatus** — enum + optional probe metadata  
- **Metadata** — произвольные key-value (без нормативных имён продуктов в схеме)

**Разрешённая цепочка диагностики:**

`Service → (LXC | VM) → Node → Cluster`

Архитектура **не содержит** встроенных знаний о конкретных продуктах; примеры в документации — только в `docs/examples/`.

## Последствия

### Положительные

- Единая модель для homelab и крупных кластеров.  
- OSS-пользователи описывают свою семантику сами.

### Отрицательные

- Service Layer требует наполнения (вручную или агентом), не из PVE.
